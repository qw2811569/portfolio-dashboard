# 客戶報告製作手冊

最後更新：2026-03-24

## 目的

這份手冊用於製作「AI + 人工分析整合版」客戶報告。目標不是把所有資訊堆滿，而是：

- 數字正確
- 語氣對客戶
- 視覺可直接交付
- 可以穩定輸出 HTML 與 PDF

適用情境：

- 持倉健檢報告
- 單一客戶投組總覽
- 人工分析 + 系統持倉整合報告
- 需要客戶版 PDF 的正式交件

## 核心原則

1. 先核對資料，再寫報告。
2. 客戶版文字只能寫結論與建議，不寫對內說明。
3. AI 只負責整合、潤稿、排版與摘要，不可自行杜撰數字。
4. 若來源彼此矛盾，先找出 authoritative source，再決定取值。
5. 若無法完全確認的數字，不要硬寫進客戶版主文。

## 必備輸入

至少需要以下兩類資料：

- `持倉資料`
  - 優先：券商匯出 PDF / 帳務 PDF / 可核對總市值與總損益的正式文件
  - 次選：完整截圖
  - 不建議直接用 app 內建 seed data
- `人工分析資料`
  - PDF / Word / 備忘錄 / 手寫整理
  - 至少要有逐檔判讀或投組調整方向

可加分資料：

- 目標價與投顧摘要
- 系統內既有 `holding dossier`
- 最新事件追蹤
- 使用者補充的口頭偏好

## authoritative source 順序

客戶報告的數字優先序固定如下：

1. 客戶持倉 PDF / 券商正式報表
2. 使用者明確指定的人工修正數字
3. 可交叉驗證的市場資料 API
4. App 內當前 state
5. 舊報告 / seed data

重點：

- `總成本 / 總市值 / 未實現損益 / 報酬率` 以正式持倉 PDF 為主
- 若個股列與總表互相矛盾，總表優先
- 若個別欄位彼此無法同時成立，不要為了排版硬湊新數字

## 標準流程

### 1. 收資料

先確認：

- 客戶名稱
- 報告名稱
- 持倉基準日
- 交付格式：HTML / PDF / 兩者都要
- 是否為「客戶版」而非「內部備忘錄」

### 2. 建立原始資料整理

在 `data/` 內建立這次案子的中間稿，例如：

- `data/<slug>-raw-notes.md`

至少整理出：

- 投組總成本
- 投組總市值
- 未實現損益與報酬率
- 每檔股票：代碼、名稱、市值、權重、損益、報酬率
- 人工判讀
- 建議動作

### 3. 先做數字校對

在寫任何 HTML 前，先逐項確認：

- 個股市值加總是否接近總市值
- 個股權重是否合理
- 主要獲利與主要拖累是否與總損益方向一致
- 報表中的極端值是否來自資料不一致

若出現矛盾，先在 raw notes 註記，不要直接寫進客戶版。

### 4. 再做結構整理

客戶版標準結構建議：

- 首頁摘要
- 投組摘要
- 配置與集中度
- 調整優先序
- 個股整理表
- 目標價或關鍵標的專區
- 未來 1–3 個月執行順序

### 5. 文字改寫成客戶版

客戶版語氣規則：

- 寫「目前結構」「後續建議」「風險觀察」
- 不寫「這次檢視」「本次我幫你整理」「人工健檢說」「系統內已收錄」
- 不寫像模型在解釋自己來源的話
- 不用口語填充，例如「重點不是...而是...」「這一區先不要」

要像：

- 正式顧問報告
- 券商簡報摘要
- 客戶會議材料

不要像：

- 對使用者的旁白
- 模型工作紀錄
- prompt 的翻譯

### 6. 產出 HTML

建議做法：

- 直接用上一份成功交付的報告 HTML 當視覺模板
- 另存成新的案件檔名
- 先完成 HTML，再輸出 PDF

### 7. 輸出 PDF

建議用 Chrome headless：

```bash
'/Applications/Google Chrome.app/Contents/MacOS/Google Chrome' \
  --headless=new \
  --disable-gpu \
  --print-to-pdf='/abs/path/output.pdf' \
  'file:///abs/path/report.html'
```

### 8. 做最終 QA

交付前至少檢查：

- 首頁四大數字是否正確
- 正負損益方向是否正確
- 權重與個股順序是否合理
- 是否有漏頁 / 空白頁 / 表格被切斷
- 是否殘留對內說明語句
- 是否有過濃的 AI 味

## 客戶版禁用句型

以下語氣預設禁止出現在客戶版：

- 這次檢視主軸
- 人工健檢指出
- 系統內已收錄
- 本報告依據...重整而成
- 問題不在...而在...
- 使用方式
- 這不是保證價位
- 不列為這次檢視主軸
- 我幫你
- 以下是我整理的

替代方向：

- 目前結構顯示
- 後續建議
- 可作為參考區間
- 對整體績效影響有限
- 後續以...追蹤為主

## AI 與人工分工

### AI 適合做

- 整合多來源文字
- 生成初版章節結構
- 將人工判讀改寫成正式文字
- 依資料生成摘要、表格、重點結論
- 協助 HTML 視覺排版
- 幫忙找出不自然或過於 AI 味的句子

### 人工一定要做

- 確認總損益與報酬率
- 確認每檔建議是否符合原始判斷
- 決定哪些資訊能對客戶公開
- 刪除內部說明
- 最後閱讀整份 PDF

## 常見錯誤

### 1. 吃到 seed data

症狀：

- 報告數字看起來完整，但和客戶真實帳面不一致

避免方式：

- 永遠先找正式持倉 PDF
- 不可直接相信 app 初始資料

### 2. 把推論寫成事實

症狀：

- 目標價、財報、邏輯寫得很滿，但沒有來源

避免方式：

- 沒來源就寫條件式
- 不要為了好看硬補數字

### 3. PDF 與 HTML 落差太大

症狀：

- HTML 好看，PDF 斷頁、淡字、空白頁

避免方式：

- 做 print CSS
- 表格允許跨頁
- 不讓整個卡片強制同頁

### 4. 語氣太像 AI

症狀：

- 看起來像在交代製作過程
- 句子太平均、太完整、太解釋

避免方式：

- 改寫成結論式短句
- 刪掉 meta 說明
- 用顧問式、簡報式語氣

## 建議交付物

每次完成後至少保留：

- `data/<slug>-client-report.html`
- `data/<slug>-client-report.pdf`
- `data/<slug>-raw-notes.md`

若要對外靜態開啟，再同步一份到：

- `public/reports/<slug>-client-report.html`

## 下次執行順序

之後再做同類型客戶報告時，照以下順序即可：

1. 先收 authoritative holdings source
2. 再收人工分析 source
3. 建 raw notes
4. 先核數字，再寫報告
5. 用正式客戶語氣，不保留對內說明
6. 先完成 HTML，再輸出 PDF
7. 最後逐頁檢查數字、語氣、斷頁與空白頁
